# 생존 영어 회화 챕터 재구성 설계서

- 작성일: 2026-06-09
- 대상 카테고리: `data/survival-english.json` (생존 영어 회화)
- 선행 문서: `2026-06-08-survival-english-30-chapters-design.md` (현 30챕터 구성)

## 1. 배경 / 문제

현 30챕터 구성을 진단한 결과, 여행 상황 커버리지가 불균형하다.

| 도메인 | 현재 챕터 | 비중 | 문제 |
|---|---|---|---|
| 항공·공항 (ch01~10) | 10개 | 33% | 여행 전체에서 하루 남짓인 출입국에 1/3 할당. 체크인/보안/게이트/기내가 잘게 쪼개져 중복 |
| 숙소 (ch11~14) | 4개 | 13% | 적정 |
| 교통 (ch15~18) | 4개 | 13% | 렌터카·교통패스·통신 누락 |
| 식음료 (ch19~21) | 3개 | 10% | 알레르기/식단제한·예약 누락 |
| 쇼핑 (ch22~24) | 3개 | 10% | 적정 |
| 관광 (ch25~27) | 3개 | 10% | 적정 |
| 금융 (ch28) | 1개 | 3% | 적정 |
| 안전·의료 (ch29~30) | 2개 | 7% | 가족 여행 핵심인데 과소 |

핵심 문제 3가지:
1. **항공/공항 과대표집** — 일회성 이벤트가 전체의 1/3.
2. **생존 기초 표현 통째 누락** — "못 알아들었어요/천천히요/다시 한번", 숫자·가격·시간 청취 등 모든 상황의 토대가 없음.
3. **현지 일상·돌발·가족 상황 빈약** — 미아, 통신(유심), 세탁, 환불/불만, 아이 동반 상황 등.

## 2. 설계 결정

- **배열 원칙**: 여행 시간순 유지. 단 맨 앞에 "생존 기초(Phase 0)" 묶음을 두어 토대를 먼저 학습.
- **범위**: 아래 9개 Phase의 도메인 전체 포함 (사용자 승인 "전체 다 좋음").
- **갯수**: 30 → **40챕터**. 항공 통합으로 비중을 낮추고, 기초·안전·일상 돌발을 확충.
- **콘텐츠 포맷**: 모든 챕터를 **ch01의 대화 중심 포맷**으로 작성한다 (아래 §2.1). 구버전 `sections` 포맷은 신규 작성 시 사용하지 않는다.
- **ch01 보존**: 사용자와 함께 다듬은 첫 공항 챕터(항공권·온라인 체크인 확인)는 **포맷 레퍼런스(골든 템플릿)로 보존**한다. 그 내용·형식을 신규 "04. 체크인·수하물" 챕터의 토대로 계승한다.

### 2.1 콘텐츠 포맷 템플릿 (ch01 기준)

각 챕터는 다음 구조를 가진다 (렌더링은 `js/chapter.js`가 `dialogues` 우선 처리, `sections`는 구버전 호환용):

- `id`, `title`, `summary`
- `dialogues[]` — **챕터당 약 10개**의 짧은 대화. 각 대화:
  - `id`, `title`
  - `lines[]` — 약 4줄, `speaker`(나/직원 등)·`en`·`ko`. (화면에서 영어 클릭 시 한국어 토글)
  - `keyExpressions[]` — 약 3개. 각 표현: `en`, `ko`(설명·혼동 주의 포함), `alternatives[]`(대체표현 약 2개, 각 `en`/`ko`)
- `sets[]` — 세트 배열. 세트 = `id` + `questions[]`. 문제는 `multiple_choice`(보기 4개) 또는 `ox`. ch01은 **세트당 15문항**(기존 10 권장보다 풍부).
- 문제 `id` 형식: `se_chNN_setMM_XXX`.

비중 변화: 항공·공항 33%(10개) → 12.5%(5개) · 안전·의료 7% → 12.5%(5개) · 기초/일상 돌발 신규 13개.

## 3. 최종 챕터 구성 (40챕터)

표기: `유지`=기존 유지, `통합`=기존 다수 챕터 병합, `신규`=신규 작성.

### 🧱 Phase 0 — 생존 기초 `신규 묶음`
- 01. 안 통할 때 버티기 — 되묻기·"천천히요"·"못 알아들었어요"·받아쓰기 요청 `신규`
- 02. 정중한 부탁·감사·사과 기본형 — Could you~ / Thank you / Sorry `신규`
- 03. 숫자·가격·시간·날짜 알아듣기 — 청취 토대 `신규`

### ✈️ Phase 1 — 출국 `10개→4개 통합`
- 04. 체크인·수하물 (온라인+카운터) `통합`
- 05. 보안검색·출국심사 `통합`
- 06. 탑승 게이트·기내 (방송·주문·요청·돌발) `통합`
- 07. 경유·환승 `유지`

### 🛬 Phase 2 — 입국
- 08. 입국심사·수하물·세관 `통합`
- 09. 공항에서 시내로 `유지`

### 🏨 Phase 3 — 숙소
- 10. 호텔/에어비앤비 체크인 `통합`
- 11. 객실 시설·요청·트러블 (와이파이/온수 고장·소음) `유지/확장`
- 12. 체크아웃 (영수증·짐보관·청구오류) `유지/확장`

### 🚇 Phase 4 — 이동·생활 셋업
- 13. 통신: 유심/이심·데이터·와이파이 `신규`
- 14. 길 묻기 `유지`
- 15. 지하철·버스 (티켓 발권기·교통패스) `유지/확장`
- 16. 기차 여행 `유지`
- 17. 택시·라이드앱 `유지`
- 18. 렌터카·주유·주차 `신규`

### 🍽️ Phase 5 — 식음료
- 19. 식당 주문 `유지`
- 20. 알레르기·식단제한·아이 메뉴 `신규`
- 21. 결제·팁·요청 `유지`
- 22. 카페·패스트푸드·포장 `유지`
- 23. 예약 전화·노쇼 확인 `신규`

### 🛍️ Phase 6 — 쇼핑·생활
- 24. 마트·장보기 `유지`
- 25. 쇼핑·옷가게 `유지`
- 26. 환불·교환·하자 항의 `신규`
- 27. 세탁 (코인세탁·호텔세탁) `신규`
- 28. 면세·택스리펀 `유지`

### 🎟️ Phase 7 — 관광·액티비티
- 29. 관광지·티켓 구매 (가족권·할인) `유지`
- 30. 박물관·투어 (촬영·오디오가이드) `유지`
- 31. 종교시설 드레스코드·매너 (바티칸/성당) `신규`
- 32. 놀이공원·가족 액티비티 `신규`
- 33. 예약 확인·취소·환불 (투어/공연) `신규`
- 34. 사진 부탁·스몰토크 `유지`

### 💳 Phase 8 — 금융
- 35. 환전·ATM·카드·DCC 거절 `유지`

### 🚨 Phase 9 — 안전·의료·위급 `2개→5개 확장`
- 36. 약국 OTC (증상·감기약) `유지`
- 37. 병원·진료 (아이 환자·처방전) `신규`

- 39. 분실·도난 신고 (경찰·대사관) `유지`
- 40. 긴급 도움 요청 (Call police/ambulance·화재·사고) `신규`

## 5. 구현 단위 / 산출물

- 단일 산출물: `data/survival-english.json` 재구성 (40챕터, 각 챕터에 `sections` + `sets`).
- 코드 변경 없음 (JSON만 편집하면 동작 — README의 무한 확장 원칙).
- 챕터 id는 `ch01`~`ch40`. 문제 id 형식 `se_chNN_setMM_XXX` 유지.

## 6. 비범위 (Out of Scope)

- 사이트 코드(JS/CSS/HTML) 변경.
- 다른 카테고리(city-info, culture, finance, packing) 변경.
- PIN/타이머/출제 로직 변경.

## 7. 열린 항목

- 각 챕터의 세트 개수 목표. ch01은 15문항 1세트. 구현 계획 단계에서 챕터별 대화 수·세트 수를 확정한다.
- 기존 24개 챕터는 현재 대부분 `sections`(구버전) 포맷이다 — 재배치 시 ch01 대화 포맷으로 **재작성**할지, 콘텐츠를 옮겨 점진 전환할지를 구현 계획에서 단계화한다.
